# Android コーディング規約

## Kotlin & Android
- UI 実装には Jetpack Compose を使用してください。
- Kotlin の標準的なコーディング規約に従ってください。
- 依存関係の管理には Gradle を使用してください。

## 実装後の検証と報告ルール
コーディング作業を完了し、ユーザーに報告(または `/save` 等でタスクを完了)する前に、必ず以下のチェックをパスさせてください。

### 必須の検証チェック項目
1. **ビルド確認**: `./gradlew assembleDebug` (またはプロジェクト指定のビルドタスク) を実行し、ビルドエラーが起きないことを確認する。
2. **ユニットテスト**: `./gradlew testDebugUnitTest` (または `./gradlew test`) を実行し、すべてのテストがパスすることを確認する。
3. **静的解析・整形**: `./gradlew ktlintFormat` や `./gradlew lint` (またはプロジェクト指定の整形タスク) を実行し、警告を修正する。

- **モジュール構造への配慮**: マルチモジュール構成の場合は、ルートから全モジュールを対象に実行するか、変更のあったモジュールを特定して個別に実行してください。

いずれかのコマンドが失敗、または警告が残る場合は、問題を解決してから再確認し、すべてパスするまで繰り返してください。
全ての検証をパスしてから、ユーザーに進捗や完了を報告してください。

## 実行権限と承認

### 許可なしで実行可能な操作 (SafeToAutoRun: true)
以下のコマンドは、検証のためにユーザーの承認なしに実行できます：
- `./gradlew assembleDebug`
- `./gradlew testDebugUnitTest`
- `./gradlew ktlintFormat`
- `./gradlew ktlintCheck`
- `./gradlew koverXmlReport`

## 解析・デバッグ時の一時ファイル
- **ログ・解析データの出力**:
  - `gradlew` の詳細ログや、エージェントが解析のために出力する一時ファイルは、リポジトリ直下の `temp/` ディレクトリ内に配置してください。
  - **後処理**: 解析が終了し、用が済んだファイルは **直ちに削除** してください。リポジトリのクリーンさを保つため、一時的なログファイルを残したままにしないでください。
