# タスク・進捗管理ルール

このプロジェクトにおける開発ライフサイクルと、ツール(roadmap.md, Issue, /save, /resume)を用いた進捗管理の手順です。

## 1. デザイン原則
- **全体俯瞰**: `docs/99_progress/roadmap.md` でプロジェクト全体の大きな流れと完了履歴を管理する。
- **実行管理**: 各タスクの具体的なステップ、現在地、および詳細なメモは **GitHub Issue** で管理する。
- **継続性**: `/save` でコンテキストを保存し、`/resume` で効率的に再開できるようにする。
- **後処理**: 作業完了後は `/cleanup` コマンドで環境をクリーンアップする。

## 2. 開発ライフサイクル (Standard Workflow)

### 2-1. 準備フェーズ：全体の把握と課題の特定
1. **ロードマップの確認**: `docs/99_progress/roadmap.md` を読み、現在のフェーズと次に着手すべき項目を確認する。
2. **Issue の作成/選択**: 着手する項目の Issue を作成(または既存のものを選択)する。
   - Issue の本文には、実装のゴールと詳細なタスクリストを記載する。

### 2-2. 着手フェーズ：ブランチ作成
1. `gh issue develop <Issue番号> --name "<Issue番号>-description" --checkout` を実行し、Issue に紐付いた作業ブランチを作成する。
   - 命名規則の詳細は [Git 運用ルール](./02_git.md#3-ブランチ作成) を参照。
2. **`working` ラベルの付与**: 対象の Issue に `working` ラベルを付与する(`gh issue edit <Issue番号> --add-label "working"`)。
   - 他に `working` ラベルがついている Issue があれば、このタイミングで剥がす。
3. **ロードマップの更新(着手中)**: `docs/99_progress/roadmap.md` の対象タスクのチェックボックスを `[/]` に更新し、着手したことを明示する。

### 2-3. 実装フェーズ：進捗の可視化と保存
1. **Draft PR の作成**: 最初のコミットをプッシュした段階で **Draft PR** を作成し、進捗を可視化する。
2. **タスクの細分化**: Brain 側の `task.md` に、そのセッションで実行する極小単位のステップを書き出す。
3. **実装と検証**: [Android コーディング規約](./04_android.md) に基づき実装し、検証コマンドをパスさせる。
4. **栞(しおり)による保存**: 中断時や保存の際は `/save` コマンドを使用する。
   - **/save の動作**: 現在のブランチ名や進捗を Issue に記録する。詳細は [3. 中断パターンと「栞(しおり)」の扱い](#3-中断パターンと栞しおりの扱い) を参照。

### 2-4. 再開フェーズ：コンテキストの復元
1. 作業を再開する際は、`/resume` コマンドを使用する。

### 2-5. 完了フェーズ：ロードマップの更新と PR
1. **ロードマップの更新**: PR をマージ可能にする前に、`docs/99_progress/roadmap.md` を更新する。
2. **PR の最終化**: 
   - 本文の「確認事項」チェックボックスをすべて埋める。
   - Draft PR を **Ready for review** に移行する(`gh pr ready`)。
   - ユーザーにマージを依頼する。
3. **クリーンアップとラベル削除**:
   - マージ完了後、ユーザーから報告を受けたら、`/cleanup` コマンドを実行してブランチの削除や `working` ラベルの除去を一括で行う。
   - 詳細は [4. クリーンアップの自動化 (/cleanup)](#4-クリーンアップの自動化-cleanup) を参照。

## 3. 中断パターンと「栞(しおり)」の扱い

作業を中断する際は、状況に応じて以下の2つのパターンを使い分けます。

### パターンA：作業途中(実装が完了していない)
- **状況**: コーディングやデバッグの途中で、まだ PR を出せる状態ではない。
- **保存**: `/save` を実行する。
  - **エージェントの行動**: `task.md` の内容を要約(完了済みタスク、未解決の課題、次の一手)し、GitHub Issue に `**[Checkpoint: In Progress]**` とタグを付けてコメントする。
- **再開**: `/resume` で `working` ラベルのついた Issue を特定し、作業を継続する。

### パターンB：一区切り(タスク完了・次が未定)
- **状況**: PR を作成した、あるいは計画していたタスクが完了し、マージ待ちの状態。
- **保存**: `/save` を実行する。
  - **エージェントの行動**: `task.md` の更新と Draft PR への push のみを行い、Issue への完了コメント投稿は行わない。
- **後処理**: マージ完了後、`/cleanup` を実行して環境を整理する。

## 4. クリーンアップの自動化 (/cleanup)

マージ完了後に行う以下の作業を `/cleanup` で自動化します。

1. `main` ブランチへの切り替えと最新化。
2. 作業ブランチのローカル削除。
3. Issue から `working` ラベルの削除。

マージによる Issue の自動クローズ履歴が「完了の証跡」となるため、別途完了コメントを投稿する必要はありません。

## 5. 進捗管理 (Source of Truth)

1. **`docs/99_progress/roadmap.md` (永続的・全体管理)**:
   - プロジェクトの「今どこにいるか」を示す地図。
   - マージされるすべての PR に、このファイルへの更新が含まれている必要がある。
2. **GitHub Issue (実行進捗・メモ / 外部公表用)**:
   - プロジェクトの公式な記録。設計判断、大きな進捗、中断時の「栞」を残す場所。
   - ユーザー(人間)とコンテキストを共有するためのマスター情報となる。
3. **Brain 側の `task.md` (一過性・細部タスク / 内部作業用)**:
   - エージェントのプライベートな作業用バッファ。
   - 1セッション内での極めて細かい作業チェックリストや、一時的な試行錯誤を記録する場所。
