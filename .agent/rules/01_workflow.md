# タスク・進捗管理ルール

このプロジェクトにおける開発ライフサイクルと、ツール(roadmap.md, Issue, /save, /resume)を用いた進捗管理の手順です。

## 1. 基本原則
- **全体俯瞰**: `docs/status/roadmap.md`（またはプロジェクト指定の場所）で全体の流れと履歴を管理する。
- **実行管理**: 各タスクの具体的なステップ、現在地、および詳細なメモは **GitHub Issue** で管理する。
- **継続性**: `/save` でコンテキストを保存し、`/resume` で効率的に再開できるようにする。
- **後処理**: 作業完了後は `/cleanup` コマンドで環境をクリーンアップする。

## 2. 開発ライフサイクル (Standard Workflow)

### 2-1. 準備・着手フェーズ：タスクの特定とセットアップ
> [!IMPORTANT]
> 開発を開始する際は、必ず Skill `kickoff-task` を `view_file` で読み込み、その手順に従ってください。

1. **ロードマップと Issue の確認**: 全体の進捗を確認し、着手する Issue を特定または作成します。
2. **作業環境の構築**: Issue に紐付いた作業ブランチを作成し、ロードマップを着手中に更新します。
   - 詳細は Skill `kickoff-task` に集約されています。

### 2-3. 実装フェーズ：進捗の可視化と保存
1. **Draft PR の作成**: 最初のコミットをプッシュした段階で **Draft PR** を作成し、進捗を可視化する。
2. **タスクの細分化**: AI 側の `task.md` に、そのセッションで実行する極小単位のステップを書き出す。
3. **実装と検証**: 各プロジェクトのコーディング規約に基づき実装し、検証コマンドをパスさせる。
4. **栞(しおり)による保存**: 作業の中断や進捗を記録したい場合、**ユーザーが `/save` コマンドを実行**し、AI が現在のコンテキストを Issue に「栞」として保存する。

### 2-4. 再開フェーズ：コンテキストの復元
1. 作業を再開する際は、**ユーザーが `/resume` コマンドを実行**する。

### 2-5. 完了フェーズ：PR 最終化とクリーンアップ
> [!IMPORTANT]
> このフェーズの開始時に、必ず Skill `wrapup-task` を `view_file` で読み込み、その手順に従ってください。

1. **成果の最終化**: ロードマップを完了に更新し、PR を Ready for review に移行します。
2. **クリーンアップ**: マージ完了後、`/cleanup` を実行して環境を整理します。
   - 詳細は Skill `wrapup-task` に集約されています。

## 6. ディレクトリ・ファイル命名規則

プロジェクト全体の整合性を維持するため、以下の命名規則に従ってください。

- **基本原則**: 原則として `kebab-case` (小文字、ハイフン区切り) を使用します。
  - 例: `user-interface/`, `setup-labels.ps1`
- **ドキュメントの順序管理**: 読み進める順序が重要なファイルやディレクトリには、`NN_` (2桁の数字とアンダースコア) のプレフィックスを付与します。
  - 例: `.agent/rules/01_workflow.md`, `02_git.md`
- **ディレクトリの役割**:
  - `docs/status/`: プロジェクトの現在の状態、ロードマップ、進捗履歴を格納します。
  - `scripts/windows/`, `scripts/linux/`: プラットフォーム別のスクリプトを、OS名を省略せずに格納します。

## 3. 中断パターンと「栞(しおり)」の扱い

作業を中断する際は、状況に応じて以下の2つのパターンを使い分けます。

### パターンA：作業途中(実装が完了していない)
- **状況**: コーディングやデバッグの途中で、まだ PR を出せる状態ではない。
- **保存**: `/save` を実行する。
  - **エージェントの行動**: `task.md` の内容を要約し、GitHub Issue に `**[Checkpoint: In Progress]**` とタグを付けてコメントする。
- **再開**: `/resume` で PR 紐付けやブランチ名から対象 Issue を自動特定し、作業を継続する。

### パターンB：一区切り(タスク完了・次が未定)
- **状況**: PR を作成した、あるいは計画していたタスクが完了し、マージ待ちの状態。
- **保存**: `/save` を実行する。
  - **エージェントの行動**: `task.md` の更新と Draft PR への push のみを行い、Issue への完了コメント投稿は行わない。
- **後処理**: マージ完了後、`/cleanup` を実行して環境を整理する。

## 4. 進捗管理 (Source of Truth)

1. **ロードマップファイル (永続的・全体管理)**:
   - プロジェクトの地図。マージされるすべての PR に、このファイルへの更新が含まれている必要がある。
2. **GitHub Issue (実行進捗・メモ / 外部公表用)**:
   - プロジェクトの公式記録。設計判断、進捗、中断時の「栞」を残す場所。
3. **AI 側の `task.md` (一過性・細部タスク / 内部作業用)**:
   - エージェントのプライベートな作業用バッファ。1セッション内での極めて細かい作業を記録する。

## 5. 共通基盤へのフィードバック

プロジェクト固有のカスタマイズの過程で、**「汎用的に使える優れたルールや改善」**を発見した場合は、積極的に共通リポジトリ (`antigravity-workflow`) へ還元してください。

- **手順**: 共通リポジトリ側に Issue を作成し、改善内容を提案する。または AI に Issue 作成を依頼する。
- **注意**: 還元する際は、プロジェクト固有の固有名詞やパスを汎用的な表現に差し替えること。
