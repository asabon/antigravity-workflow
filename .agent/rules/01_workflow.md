# タスク・進捗管理ルール

このプロジェクトにおける開発ライフサイクルと、ツール(roadmap.md, Issue, /dev-pause, /dev-start)を用いた進捗管理の手順です。

## 1. 基本原則
- **全体俯瞰**: `docs/status/roadmap.md`（またはプロジェクト指定の場所）で全体の流れと履歴を管理する

## 2. 開発ライフサイクル (Standard Workflow)

開発は「**セッション状態 (Session State)**」と「**フェーズ (Phase)**」の2軸で進行します。

### 2-0. セッション状態と状態遷移表

エージェントの活性状態とライフサイクルの関係は以下の通りです。

| 現在の状態 | 現在のフェーズ | トリガー（イベント） | 🚀 次の状態 | 🚀 次のフェーズ | 備考 / アクション |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **`IDLE`** | **`-`** | **セッション開始** / `/setup` | `ACTIVE` | `PLANNING` | ロードマップ確認から開始する |
| **`IDLE`** | **`-`** | **`/dev-start`** | `ACTIVE` | *(自動判定)* | `01_workflow.md` 2-5 の自動判定による |
| **`ACTIVE`** | `PLANNING` | **対象 Issue 等の確定** | `ACTIVE` | `PREPARING` | `flow-kickoff` の実行へ移行 |
| **`ACTIVE`** | `PREPARING` | **`flow-kickoff` 完了** | `ACTIVE` | `DEVELOPING` | Draft PRが作成され、実装を開始 |
| **`ACTIVE`** | `DEVELOPING` | **`flow-wrapup` 完了** | `ACTIVE` | `REVIEWING` | 自動検証パス、PR が Ready 状態 |
| **`ACTIVE`** | `REVIEWING` | **マージ完了 ＆ `/cleanup`** | `ACTIVE` | `PLANNING` | 次のタスク計画、または「次どうする？」の相談へ戻る |
| **`ACTIVE`** | `REVIEWING` | **ユーザーからの指摘（要修正）** | `ACTIVE` | **`DEVELOPING`** | 既存ブランチで修正。**新規 Issue 作成は厳禁** |
| **`ACTIVE`** | *Any* | **`/dev-pause` （中断・栞）** | **`IDLE`** | **`-`** | 栞を残してエージェントが待機（セッション中断） |

---

### 2-1. 計画フェーズ (PLANNING)：タスクの選定
- **目的**: 次に着手するタスク（Issue）を確定、またはアイデアの掘り下げを行う。
- **[開始条件]**: 新規セッション、`/cleanup` 完了直後、**または `docs/status/backlog.md` に要望が追記された時。**
- **[完了と遷移]**: 対象 Issue が確定し、`roadmap.md` の対象項目に Issue 番号が紐付いた時点で、自動的に **`PREPARING` 状態へ遷移** します。

**💡 運用手順 (バックログ連携)**:
1. **バックログの点検**: `PLANNING` フェーズに入ったエージェントは、まず `docs/status/backlog.md` の「📥 未分類の要望」がないか確認します。
2. **タスク分解と反映**: 未分類の要望がある場合、要件の掘り下げやタスク分解を行い、ドラフト（案）として `docs/status/roadmap.md` の適切なフェーズへ転記・提案します。
3. **ステータスの更新**: 転記した項目は、`backlog.md` 内で「📋 検討中」または「✅ 反映済」へステータスを移動させてください。

**📜 通常の手順**:
1. **ロードマップの確認**: `docs/status/roadmap.md` を読み、全体のゴールと進捗を確認します。
2. **次の一手の提案**: 未着手の一覧から優先順位を判断し、ユーザーに提案します。
3. **Issue の確定**: ユーザーがタスクを選択したら、既存の Issue を読むか、ない場合は新規起票します。
   - **同期**: 確定した Issue 番号を `roadmap.md` の対象項目に付与（例: `- [ ] #123 タイトル`）します。

### 2-2. 準備・着手フェーズ (PREPARING)：環境セットアップ
> [!IMPORTANT]
> このフェーズを開始する際は、必ず **Skill `flow-kickoff`** を `view_file` で読み込み、その自動化手順（ブランチ作成、Draft PR作成、ロードマップ更新等）を漏れなく実行してください。

- **[開始条件]**: PLANNING 完了直後、または `/dev-start` 時に準備不足を検知した場合。
- **[完了と遷移]**: **`flow-kickoff`** の手順がすべて完了した時点で、自動的に **`DEVELOPING` 状態へ遷移** します。


### 2-3. 実装フェーズ (DEVELOPING)：実装ループ
> [!IMPORTANT]
> 開発を完了し、PR を最終化する際は、必ず **Skill `flow-wrapup`** を `view_file` で読み込み、その手順に従ってください。

- **[開始条件]**: PREPARING 完了直後、または `/dev-start` 時に実装中であることを検知した場合。
- **[完了と遷移]**: **`flow-wrapup`** の最終手順がすべて完了した時点で、自動的に **`REVIEWING` 状態へ遷移** します。

1. **実装と検証の反復**: `task.md` に基づき実装し、都度テストや検証コマンドを実行して品質を確保します。
2. **進捗の可視化**: 進捗はAI側の `task.md` を更新しながら進めます。
3. **栞(しおり)による保存**: 作業の中断や状態保存が必要な場合、**ユーザーまたはAI主導で `/dev-pause` コマンドを実行**し、Issue に Checkpoint コメントを残します。
4. **再開**: 中断からの復帰は **ユーザーが `/dev-start` コマンドを実行**し、文脈を自動復元します。
5. **成果の最終化**: 実装完了時は、必ず **Skill `flow-wrapup`** に従い、最終動作検証や `roadmap.md` の更新、PR の最終化を安全に実行してください。


### 2-4. レビュー・完了フェーズ (REVIEWING)：マージと後処理
- **[開始条件]**: DEVELOPING 完了直後（PR が Ready 状態）、または `/dev-start` 時にレビュー待ちを検知した場合。
- **[完了と遷移]**: マージが完了し、ユーザーから `/cleanup` 指示を受けてブランチクリーンアップが完了した時点で、自動的に **`PLANNING` 状態へ遷移**（次のタスクへ）します。

1. **ユーザーレビューの待機と対応**: ユーザーのレビューを受け、指摘があれば修正（`DEVELOPING` へ）。既存の作業ブランチ・PR をそのまま使用し、**絶対に新規 Issue を作成してはいけません**。承認されればマージを待ちます。
2. **クリーンアップ**: マージ完了後、ユーザーからの `/cleanup` 指示を受け、作業ブランチをクリーンアップします（次回の PLANNING へ）。

---

### 2-5. 状態の自動判定ロジック (State Detection on Resume)

作業を再開（`/dev-start`）した際、エージェントは以下の **「環境の成果物・ステータス」から現在の状態を自律的に判定** します。

| 現在の環境ステータス | 判定される状態 (State) | 必要アクション |
| :--- | :--- | :--- |
| **`main` ブランチにいる** ＆ 作業ブランチがない | **`PLANNING`** | `roadmap.md` の確認から開始 |
| **作業ブランチはある** ＆ **PR が存在しない** | **`PREPARING`** | `flow-kickoff` の残りを実行し、Draft PR 作成 |
| **Draft PR が存在する** | **`DEVELOPING`** | `task.md` の読込、または checkpoint から実装を継続 |
| **PR が Ready (審査待ち)** 状態である | **`REVIEWING`** | ユーザーレビューまたはマージを待機（または指摘対応） |

※ 判定に迷った場合や不整合がある場合は、必ず **ユーザーへ現在の状態の確認** を行い、安全を担保します。

## 3. ディレクトリ・ファイル命名規則

### 3-1. 基本原則
原則として `kebab-case` (小文字、ハイフン区切り) を使用します。
- 例: `user-interface/`, `setup-labels.ps1`

### 3-2. 順序管理
読み進める順序が重要なファイルやディレクトリには、`NN_` (2桁の数字とアンダースコア) のプレフィックスを付与します。
- 例: `.agent/rules/01_workflow.md`, `02_git.md`

---

## 4. 中断パターンと「栞(しおり)」の扱い

作業を中断する際は、状況に応じて以下の2つのパターンを使い分けます。

### 4-1. パターンA：作業途中(実装が完了していない)
- **状況**: コーディングやデバッグの途中で、まだ PR を出せる状態ではない。
- **保存**: `/dev-pause` を実行する。
  - **エージェントの行動**: `task.md` の内容を要約し、GitHub Issue に `**[Checkpoint: In Progress]**` とタグを付けてコメントする。

### 4-2. パターンB：一区切り(タスク完了・次が未定)
- **状況**: PR を作成した、あるいは計画していたタスクが完了し、マージ待ちの状態。
- **保存**: `/dev-pause` を実行する。
  - **エージェントの行動**: `task.md` の更新と Draft PR への push のみを行い、Issue への完了コメント投稿は行わない。

---

## 5. 進捗管理 (Source of Truth)

1. **ロードマップファイル (永続的・全体管理)**:
   - プロジェクトの地図。
   - **重要**: 各項目には **一意な番号（例: `1-1`, `2-1`）をプレフィックスとして付与** し、指示の明確化やトレーサビリティを確保すること。また、対応する **GitHub Issue 番号（例: #38）** を記載すること。
2. **GitHub Issue (実行進捗・メモ / 外部公表用)**:
   - プロジェクトの公式記録。設計判断、進捗、中断時の「栞」を残す場所。
3. **AI 側の `task.md` (一過性・細部タスク / 内部作業用)**:
   - エージェントのプライベートな作業用バッファ。1セッション内での極めて細かい作業を記録する。

---

## 6. 共通基盤へのフィードバック

プロジェクト固有のカスタマイズの過程で、**「汎用的に使える優れたルールや改善」**を発見した場合は、積極的に共通リポジトリ へ還元してください。

- **手順**: 共通リポジトリ側に Issue を作成し、改善内容を提案する。または AI に Issue 作成を依頼する。
- **注意**: 還元する際は、プロジェクト固有の固有名詞やパスを汎用的な表現に差し替えること。
