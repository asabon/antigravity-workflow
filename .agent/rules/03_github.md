# GitHub (Issue / PR) 運用ルール

GitHub 上で Issue や Pull Request (PR) を作成・管理する際の統一ルールです。

## 1. コンテンツの言語
- **Issue / PR**: タイトルおよび本文は、原則として**日本語**で記述してください。
- **ブランチ名**: [Git 運用ルール](./02_git.md) に従い、**英語**を使用してください。

## 2. Issue の作成
- **タイトル**: 接頭辞、カテゴリ、Issue番号などは**含めない**でください。
  - **正**: `割引計算ロジックの実装`
  - **アンチパターン**: `【機能追加】割引計算ロジックの実装`, `fix: パグ修正`, `#123 概要`
- **ラベル**: 必ず適切なラベル(`enhancement`, `bug`, `documentation` 等)を実際に付与してください。
- **本文**: 以下の項目を必ず含めてください。
  - 背景・目的
  - 実施内容
  - 期待される動作(または再現手順)

## 3. Pull Request (PR) の作成

### 3-1. PR タイトル
PR タイトルは、Release Drafter によって自動的にリリースノートに収集されるため、以下の規約を厳守してください。
- **言語**: 必ず**日本語**で記述してください。
- **形式**: 接頭辞(例：`feat:`, `fix:`)は**含めない**でください。カテゴリは GitHub ラベルで管理します。
- **理由**: PR タイトルは Release Drafter によってそのままリリースノートの項目として使用されるため、エンドユーザー向けの読みやすい表現にする必要があります。
- **内容**: リリースノートにそのまま載ることを意識し、簡潔かつ分かりやすい表現にしてください。
- **例**:
  - `○ 割引計算ロジックの実装`
  - `× feat: 割引計算ロジックの実装`

### 3-2. ラベルの定義と役割
ラベルは、その「目的」に応じて以下の 2 種類に分類されます。

1. **分類ラベル (Category Labels)**:
   - **用途**: Issue や PR の種類(機能追加、バグ修正、ドキュメント等)を永続的に示す。
   - **タイミング**: 作成時に付与し、基本的には変更しない。
   - **主なラベル**: `enhancement`, `bug`, `documentation`, `refactor`, `chore`, `test`
   - **リリース調整ラベル**: `major`, `minor`, `breaking` (バージョニングの制御に使用)
2. **状態ラベル (Status Labels)**:
   - **用途**: エージェントが現在「どのタスクに注力しているか」を一目で識別するための目印。
   - **タイミング**: 作業開始時に付与し、**完了時(マージ後の /cleanup)に剥がす**。中断時(/save)は、後で `/resume` できるように**付与したまま**にする。
   - **主なラベル**: `working`

### 3-3. 本文 (Description)
PR の本文には、以下のセクションを必ず含めてください。
- **内容 (Summary)**: 変更の要約。対応する Issue がある場合は、`Closes #<Issue番号>` や `Resolves #<Issue番号>` を含めて自動クローズ連携させてください。
- **変更点 (Changes)**: 具体的な修正内容の箇取り
- **確認事項 (Verification)**: 実施したテストや動作確認の結果。**チェックボックス (`- [ ]`) を活用**し、エージェントはすべての準備が整った段階でこれらを `[x]` に更新した上でユーザーに報告してください。
- **ラベル**: 付与したラベルをここにも記載してください。

### 3-4. Draft PR の活用
- **原則**: 作業を開始して最初のプッシュを行った段階で、**Draft PR** として作成することを推奨します。
  - `gh pr create --draft`
- **移行**: すべての作業が完了し、本文の「確認事項」にすべてチェックが入った段階で **Ready for review** に移行し、ユーザーにマージを依頼してください。

### 3-5. マージ戦略
- **推奨戦略**: 原則として **Squash and merge** を使用してください。
- **コミットメッセージ**:
  - Squash マージ時のコミットメッセージには、**PR タイトル** をそのまま使用することを推奨します。
  - これにより、`main` ブランチの履歴がクリーンに保たれ、リリースノートの生成がスムーズになります。

## 4. 実行権限と承認

### 厳禁事項 (絶対に実行しないでください)
- **`main` ブランチへの直接 push**
- **Pull Request の承認 (Approve) および マージ (Merge)**
  - これらは最終的な品質担保として、必ずユーザー(人間)が行います。

### 許可が必要な操作 (SafeToAutoRun: false)
- **Issue / PR の作成と編集**: `gh issue create / edit`, `gh pr create / edit` (意図しない情報公開を防ぐため)
- **コミュニケーション**: `gh issue comment`(進捗報告やお知らせの投稿)
- **リポジトリ設定の変更**: `gh label create / edit` など。
- **破壊的な操作**: コメントの削除や Issue のクローズ(`Closes` による自動クローズを除く)。

### 許可なしで実行可能な操作 (SafeToAutoRun: true)
- **情報の取得**: `gh issue view / list`, `gh pr view / list`, `gh label list`, `gh repo view`
- **認証・状態確認**: `gh auth status`, `gh --version`

## 5. 技術的トラブルへの対応 (CLI)

CLI 操作(特に `gh` や `git`)において、文字欠けやエスケープミスによる事故を防ぐためのガイドラインです。

- **多行・長文テキストの扱い (一時ファイル)**:
  - `gh pr create --body` や `git commit -m` に直接長い文字列を渡すと、エスケープミスや文字化けの原因になります。
  - **原則**: リポジトリ直下の `temp/` ディレクトリ内に一時ファイルを生成し、`--body-file <ファイル名>` 等を使用して読み込んでください(`temp/` は `.gitignore` で保護されています)。
  - **後処理**: 作業(コマンドの実行)が完了したら、生成した一時ファイルは **直ちに削除** してください。リポジトリ内に不要なファイルを残さないようにします。
- **入力ミス(先頭文字の欠落)の防止**:
  - 実行環境によってはコマンドの先頭文字が無視されることがあります。
  - **回避策**: `& gh ...` のように呼び出し演算子を使用するか、コマンド実行直後に `gh pr view` 等で意図通り反映されたかを確認してください。
