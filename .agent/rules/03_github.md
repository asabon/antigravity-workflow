# GitHub (Issue / PR) 運用ルール

GitHub 上で Issue や Pull Request (PR) を作成・管理する際の統一ルールです。

> [!CAUTION]
> ### 絶対禁止事項 (Zero Tolerance)
> 以下の行為は、いかなる場合も厳禁です。
> 1. **タイトルへのプレフィックス混入**: `feat:`, `fix:` 等をタイトルに入れないでください（日本語のみ）。
> 2. **Draft 状態での放置**: レビューやマージを依頼する前に、必ず `gh pr ready` を実行してください。
> 3. **テンプレートの回避**: `gh` コマンドの `--body` オプション等でテンプレートを無視しないでください。

## 1. GitHub flow の徹底
本リポジトリは GitHub flow に準拠します。
1. **まず Issue を立てる**: 作業を開始する前に、必ず Issue で意図を明確にしてください。
2. **早期に Draft PR を出す**: 最初のコミットをプッシュした直後に Draft PR を作成し、透明性を確保してください。

## 2. コンテンツの言語
- **Issue / PR**: タイトルおよび本文は、必ず**日本語**で記述してください。
- **ブランチ名**: [Git 運用ルール](./02_git.md) に従い、**英語**を使用してください。

## 3. Issue の作成
Issue を作成・更新する際は、必ず **Skill `github-issue`** を使用してください。
このスキルは、タイトル検証や安全な作成（一時ファイル利用）、コメント追加などを統合管理します。

- **ラベル**: 作成時に必ず適切な**分類ラベル**（セクション 5 参照）を付与してください。

## 4. Pull Request (PR) の作成と管理
PR を作成・管理する際は、必ず **Skill `github-pr`** を使用してください。
このスキルは、Draft 作成から本文更新、Ready for review（最終化）までを統合管理します。

- **Draft 活用**: 作業開始後の最初のプッシュ時に、必ず **Draft PR** 状態であることを確認してください。
- **タイトル**: プレフィックス（`feat:`, `fix:` 等）は**含めない**でください。
- **ラベル**: 作成時に必ず適切な**分類ラベル**を付与してください。
- **本文 (Description)**: 関連 Issue の紐付け（`Closes #<Issue番号>`）を必ず含めてください。

## 4. ラベルの定義と運用
ラベルは GitHub 上での視認性と、リリースノート生成（Release Drafter）の制御に使用します。

1. **分類ラベル (必須)**:
   - `enhancement`: 機能追加
   - `bug`: バグ修正
   - `documentation`: ドキュメント関連
   - `refactor`: リファクタリング
   - `chore`: 雑用、ビルドプロセス、補助ツールの変更など
   - `test`: テストの追加や修正
   - `research`: 調査、リサーチ、技術検証
   - `question`: 質問、確認事項
   - `discussion`: 議論、検討、設計相談
   - `security`: セキュリティに関連する修正・報告
2. **リリース制御ラベル (単一付与)**:
   - バージョニングを伴うリリースの際、次のバージョン番号を制御するために使用します。
   - **ルール**: 1つの PR に対して、以下のうち**最大1つまで**を付与してください（重複付与はバージョン計算の誤動作を招くため禁止です）。
   - `major`: メジャーバージョンアップとなる、互換性のない大規模な変更。
   - `minor`: 新機能の追加など、互換性を維持した機能拡張（マイナーバージョンアップ）。
   - `breaking`: 既存の挙動やインターフェースを修正・変更する際の破壊的変更（警告として使用）。

## 5. マージと履歴管理
- **戦略**: 原則として **Squash and merge** を使用します。
- **コミットメッセージ**: Squash マージ時のメッセージには、**PR タイトル** をそのまま使用してください。これにより `main` の履歴がリリースノートとして読みやすくなります。

## 6. 実行権限
- **ユーザー（人間）限定**: `main` ブランチへの直接 push、および PR の承認 (Approve) / マージ (Merge)。
- **AI 実行（要承認）**: Issue/PR の作成・編集、コメント投稿、ラベル作成等の GitHub API 操作。

## 7. CLI トラブル防止 (テクニカル)
- **長文本文の投稿**: 文字化けやエスケープミスを防ぐため、長文の本文は AI の作業用内部ディレクトリ（例：Brain ディレクトリ等）など、**ワークスペース外部の一時領域** にファイルを作成し、`--body-file` オプションを使用して投稿してください。実行後は必ずファイルを削除してください。
- **入力補填**: コマンドの先頭文字欠落を防ぐため、`& gh ...` の使用や、実行後の `gh pr view` による確認を推奨します。
