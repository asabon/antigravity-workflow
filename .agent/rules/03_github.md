# GitHub (Issue / PR) 運用ルール

GitHub 上で Issue や Pull Request (PR) を作成・管理する際の統一ルールです。

## 1. コンテンツの言語
- **Issue / PR**: タイトルおよび本文は、必ず**日本語**で記述してください。
- **ブランチ名**: [Git 運用ルール](./02_git.md) に従い、**英語**を使用してください。

## 2. Issue の作成
- **タイトル**: プレフィックス（`feat:`, `fix:` 等）、カテゴリ、Issue番号は**含めない**でください。
  - **正**: `割引計算ロジックの実装`
  - **否**: `【機能追加】割引計算ロジックの実装`, `fix: パグ修正`, `#123 概要`
- **ラベル**: 作成時に必ず適切な**分類ラベル**（セクション 4 参照）を付与してください。
- **本文 (Body)**:
  - リポジトリ内の `.github/issue_template/` 下にある適切なテンプレート（例：`task.md`）を読み込み、その構成に従ってください。
  - AI が自律的に作成する際は、テンプレートの内容に加え、**「期待される動作 (Expected Behavior)」** が明確であることを確認（不足していれば追記）してください。

## 3. Pull Request (PR) の作成
- **Draft 活用**: 作業開始後の最初のプッシュ時に、必ず **Draft PR** (`gh pr create --draft`) として作成してください。
- **タイトル**: 
  - 必ず**日本語**で記述してください。
  - 接頭辞 (`feat:`, `fix:` 等) は**含めない**でください（カテゴリはラベルで管理します）。
- **ラベル**: 作成時に必ず適切な**分類ラベル**を付与してください。
- **本文 (Description)**: 以下の要素を必ず含めてください。
  - **内容 (Summary)**: 変更の要約。
  - **Issue 紐付け**: 関連する Issue がある場合、`Closes #<Issue番号>` を含めてください。これにより GitHub の Development セクションに自動リンクされ、`/resume` の精度が向上します。
  - **変更点 (Changes)**: 具体的な修正内容の箇条書き。
  - **確認事項 (Verification)**: 実施したテスト結果。チェックボックス (`- [ ]`) を用意し、完了したものから `[x]` に更新してください。
- **PR の最終化**: 
  - 本文の「確認事項」がすべて完了した段階で **Ready for review** (`gh pr ready`) に移行し、ユーザーに報告してください。

## 4. ラベルの定義と運用
ラベルは GitHub 上での視認性と、リリースノート生成（Release Drafter）の制御に使用します。

1. **分類ラベル (必須)**: `enhancement` (機能追加), `bug` (バグ修正), `documentation` (ドキュメント), `refactor` (リファクタ), `chore` (雑用), `test` (テスト)
2. **リリース制御ラベル**: `major`, `minor`, `breaking` (バージョニングが必要な場合に使用)

## 5. マージと履歴管理
- **戦略**: 原則として **Squash and merge** を使用します。
- **コミットメッセージ**: Squash マージ時のメッセージには、**PR タイトル** をそのまま使用してください。これにより `main` の履歴がリリースノートとして読みやすくなります。

## 6. 実行権限
- **ユーザー（人間）限定**: `main` ブランチへの直接 push、および PR の承認 (Approve) / マージ (Merge)。
- **AI 実行（要承認）**: Issue/PR の作成・編集、コメント投稿、ラベル作成等の GitHub API 操作。

## 7. CLI トラブル防止 (テクニカル)
- **長文本文の投稿**: 文字化けやエスケープミスを防ぐため、長文の本文は `temp/` 下に一時ファイルを作成し、`--body-file` オプションを使用して投稿してください。実行後は一時ファイルを削除してください。
- **入力補填**: コマンドの先頭文字欠落を防ぐため、`& gh ...` の使用や、実行後の `gh pr view` による確認を推奨します。
