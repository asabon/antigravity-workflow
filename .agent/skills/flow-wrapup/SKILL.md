# flow-wrapup Skill

## 目的 (Purpose)
作業完了時のユーザーへの報告、および Pull Request (PR) 発行・最終化までの手順を明確にし、**「成果物を安全かつ確実に main ブランチへ届ける」**ことを目的とします。

## 1. 最終動作検証
- PR を出す前に、すべての関連テストやビルドがローカル環境でパスすることを確認します。
- **セルフレビュー**: 変更したファイルの一覧と差分を `git diff` で確認し、意図しない変更（デバッグ用の出力など）が残っていないかチェックしてください。
- `walkthrough.md` に記載するエビデンス（ログ、スクリーンショット等）を取得します。

## 2. PR 本文の最終チェック
- `gh pr view --json body` で現在の内容を確認します。
- **禁止事項**: タイトルに `feat:` 等が含まれていないか再確認します。
- **必須**: `Closes #<Issue番号>` が含まれているか確認します。
- テンプレート内のチェックボックスを、実際の状況に合わせて更新します。

## 3. ロードマップの完了更新
- `docs/status/roadmap.md` を開き、対象タスクを完了（`- [x]`）としてマークします。
- **重要**: この更新を `git commit` および `git push` し、PR に含めてください。

## 4. PR を Ready for review にする (最終化)
- **絶対条件**: 以下のコマンドを実行し、自動検証に合格（**`[RESULT] Verification PASSED!`**）することを確認します。
  `powershell -ExecutionPolicy Bypass -File .agent/skills/github-pr/scripts/verify-pr.ps1 -PrNumber <PR番号>`
  ※ 合格しない限り、次のステップに進んで不具合の修正を行ってください。
- **実行**: `gh pr ready <PR番号>` を実行し、Draft 状態を解除（Ready）します。

> [!CAUTION]
> **完了報告の絶対条件ガード (Safe Gate)**
> 上記の自動検証が「PASSED」になり、且つ PR が `Ready for review`（Draft 解除済み）になるまでは、**決して `notify_user`（完了報告）を実行してはなりません**。


## 5. ユーザーへの報告
- 実施した変更内容、検証結果、および成果物の確認を依頼します。
- **重要**: 「承認・マージが完了しましたら、`/cleanup` を実行して作業ブランチをクリーンアップいたします」とアナウンスし、事後処理への移行をスムーズにします。

> [!CAUTION]
> PR を作成・最終化する瞬間にこそ、`.agent/rules/03_github.md` の「絶対禁止事項」を今一度読み直してください。
